# AGENT.md

このプロジェクトで作業するエージェント向けの開発ルールです。

## Product Intent

このプロダクトは、転職者との面談内容をもとに、その内容にマッチする企業を外部企業データベースから選出するための補助ツールです。

企業を自動決定するものではありません。キャリアアドバイザーが確認・判断するための材料を整理します。

## Terminology

使う言葉:

- 企業を選出する
- 選出企業
- 選出区分
- 選出理由
- マッチする企業
- 面談内容との照合
- 懸念点
- 追加確認すべき質問

使わない言葉:

- 推薦企業
- 推薦
- おすすめ
- 紹介候補
- AIが決定する
- 点数
- スコア

ただし、技術的に内部処理で並び順を決めるための重み付けを使う場合でも、ユーザー向け出力には点数やスコアを表示しない。

## Selection Labels

企業選出の判断区分は以下に統一する。

- マッチ度が高い
- マッチしている
- 条件確認が必要
- 現時点ではマッチが弱い

各企業には必ず以下を含める。

- 選出区分
- 選出理由
- 懸念点
- 追加確認すべき質問

## MVP Priority

まずはテキスト入力版MVPを優先する。

```txt
面談テキスト
-> 転職者プロフィール整理
-> 企業DBと照合
-> マッチする企業候補を選出
-> 選出理由・懸念点・追加質問を表示
```

外部DB連携、リアルタイム音声入力、UIはこの後に進める。

## Implementation Rules

- 既存のディレクトリ構成を尊重する。
- 面談内容の整理は `app/core/analysis/` に置く。
- 企業との照合・選出ロジックは `app/core/matching/` に置く。
- 外部企業DB連携は `app/integrations/` 配下に分離する。
- 実行用スクリプトは `scripts/` に置く。
- サンプルデータは `data/sample/` に置く。
- ルールベース実装とLLM実装を混ぜすぎない。LLMに置き換える場合も、入出力形式は保つ。
- ユーザー向けの出力は日本語を基本にする。
- プログラミングを書くときは、そのコードが何をするためのものか分かる説明を残す。
- 複雑な処理、判断ロジック、データ変換には、処理の意味や意図が分かるコメントを付ける。
- 自明な処理にまで細かいコメントを付けすぎず、後から読む人が判断理由を理解できることを優先する。

## Data Rules

転職者プロフィールは、最低限以下の観点を整理する。

- 現職・職種
- スキル
- 希望職種
- 希望業界
- 希望勤務地
- リモート希望
- 希望年収
- キャリア志向
- 避けたい条件
- 重視する価値観

企業データは、最低限以下の観点を持つ。

- 企業名
- 業界
- 募集職種
- 必須スキル
- 歓迎スキル
- 勤務地
- リモート条件
- 年収レンジ
- カルチャー
- 募集メモ

## Safety and Privacy

- 実在の面談データや個人情報をサンプルデータに入れない。
- 個人名、電話番号、メールアドレス、住所などは保存前にマスキングする方針を前提にする。
- AIの出力は自動決定ではなく、確認材料として扱う。
- 不確かな条件は断定せず、追加確認すべき質問として返す。

## Verification

Pythonコードを変更したら、最低限以下を確認する。

```bash
python3 -m compileall app scripts
python3 scripts/run_mvp.py
```

選出結果では、以下を確認する。

- 点数やスコアが表示されていない
- 選出区分が指定ラベルのどれかになっている
- 選出理由がある
- 懸念点がある場合は追加確認すべき質問もある
- 避けたい条件に近い企業が強く扱われていない
